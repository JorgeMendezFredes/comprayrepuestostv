# MD #6 — Implementación (Astro) + estructura del proyecto + placeholders de imágenes
**Propósito:** Instruir al LLM para implementar el sitio en **Astro** como un proyecto real, listo para deploy, usando el dominio `https://www.comprayrepuestostv.cl`.  
Debe incluir:
- Rutas/páginas definidas,
- layout reutilizable,
- SEO técnico (canonical, meta, OG, JSON-LD),
- componentes de UI (CTA, cards, FAQ),
- y **placeholders visibles de imágenes** con descripción (porque aún no hay imágenes reales y probablemente se generarán con prompts luego).

> **Fuente de verdad:** MD #1 (datos), MD #2 (copy), MD #3 (metas/keywords/interlinking), MD #4 (JSON-LD/robots/sitemap), MD #5 (UI).  
> **Restricción crítica:** No inventar reseñas, certificaciones, email, redes.

---

## 0) Objetivos técnicos (OBLIGATORIOS)
1) Astro + TypeScript opcional (preferible), sin dependencias innecesarias.
2) Sitio **estático** (SSG) para performance, salvo que se requiera algo dinámico mínimo (FAQ accordion con JS liviano).
3) SEO completo:
   - Titles + meta descriptions por página
   - canonical por página
   - OG/Twitter básico
   - JSON-LD (LocalBusiness + FAQPage + BreadcrumbList)
   - robots.txt + sitemap.xml
4) UI profesional, mobile-first, CTA flotante WhatsApp y botón llamar.
5) Placeholders de imágenes:
   - Cada sección que normalmente llevaría imagen debe mostrar un bloque visible “Imagen pendiente” + una **descripción** de qué imagen va ahí.
   - Los placeholders deben ser consistentes y listos para reemplazar por `src` real cuando existan imágenes.

---

## 1) Stack y dependencias
- Framework: **Astro**
- Estilos: CSS vanilla (recomendado) o Tailwind (solo si ya está decidido; si no, usar CSS).
- Interactividad: JS mínimo (FAQ accordion, menú mobile).
- Sin CMS.
- Sin e-commerce.
- Sin librerías de UI pesadas.

Debe:
1) Setear `<html lang="es-CL">`
2) Incluir:
   - `<title>`
   - `<meta name="description">`
   - canonical
   - OG/Twitter tags (sin `og:image` si no hay imagen real)
3) Inyectar JSON-LD según página:
   - LocalBusiness: en Home y Contacto (o global si se decide, pero consistente)
   - BreadcrumbList: en páginas internas
   - FAQPage: solo si esa página renderiza FAQ visible
4) Incluir `Header`, `Footer`, `WhatsAppFloat`.
5) Aplicar CSS global + diseño sobrio según MD #5.

## 6) Componentes obligatorios (resumen de responsabilidad)
### Header.astro
- Menú simple (Inicio, Compra, Repuestos, Reparación, Productos, Contacto)
- Botón WhatsApp destacado
- Menú hamburguesa en mobile

### WhatsAppFloat.astro
- Botón flotante en mobile (y opcional en desktop)
- Link a WhatsApp usando `phoneDigits` y mensaje prellenado genérico:
  “Hola, quiero cotizar. Marca y modelo: ____.”

### CTAButtons.astro
- Renderiza 2–3 botones:
  - WhatsApp
  - Llamar (tel:)
  - Cómo llegar (anchor a bloque de dirección o /contacto)

### ServiceCard.astro
- Card con título, descripción, CTA.
- Sin inventar precios.

### Steps.astro
- 3 pasos (reutilizable).
- Debe aceptar array de pasos.

### FAQAccordion.astro
- Acordeón accesible: botones con `aria-expanded`, paneles con `id` y `aria-controls`.
- JS mínimo o details/summary nativo (preferible).
- Solo preguntas autorizadas por MD #2/MD #4.

### ImagePlaceholder.astro (CLAVE)
**Debe ser visible y útil para el equipo:**
- Un contenedor con borde punteado.
- Título: “Imagen pendiente”
- Campos:
  - `title` (qué imagen va)
  - `description` (qué debe mostrar)
  - `suggestedPrompt` (opcional): 1–2 líneas de prompt sugerido para generar la imagen posteriormente
  - `sizeHint` (ej. 1600x900 hero, 1200x800 card)
- Debe renderizarse en secciones donde normalmente iría una imagen.

Ejemplo de uso:
- Hero: placeholder “Foto del local / repuestos”
- Confianza: placeholder “Foto del equipo o mesón”
- Productos: placeholder “Foto controles/bases/patas”
